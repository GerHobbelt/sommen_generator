Create Calculus Exercise Leaf (= 'Chapter') - Documentation

This (web) tool can be used to generate unlimited series of calculus exercises for primary school children. You can either select one or more 'presets' (Use CONTROL or SHIFT key with the mouse to select multiple presets at once) in the top half of the web form here or here and/or specify your own custom exercise setup in the bottom half of the web form here.

Definition of terms used throughout this system and its documentation

Exercise

A single 'sum'.

This use of the term 'exercise' is slightly off from the usual meaning in (primary) school. In this system, 'exercise' is a single item to work on. As the basic element of this system is the 'sum' itself and not the 'sheet' of paper it is printed on or the 'task', i.e. the collection of sums of which this one sum will be but a small part, this may lead to a bit of confusion at first. See also why we use the term 'exercise' like that in the description of the term 'style' below.

Chapter

A single collection of sums (exercises); often—but not necessarily—all these exercises are of one type (a.k.a. 'style'). One 'chapter' can be thought of as one sheet of sums. Technically, 'chapters' are sequence numbers identifying different sheets of exercises of the same type (or collection of types ('styles')).

As such, a 'chapter' number does not uniquely identify the exercise sheet! (See the fourth example for a (partial) solution for this. See also the tips in the preceding examples, for example here, for other ways to cope with this.)

Think of a 'chapter' as a single task or exercise sheet.

'Chapters' are designed to be handed out to students as one-at-a-time tests: a 'chapter' is a unit of work for a student and should be completed on a single day. You may choose to hand out 'chapters' in bundles and/or request the student to complete multiple exercises ('chapters') on a single day; that is up to you and the situation. However, a 'chapter' should be completed, once started.

Style (a.k.a. 'type')

An exercise definition, which is used by the generator to produce/select a series of exercises of this particular type. A style, for instance, instructs the generator to produce exercises to help the student exercise a particular subject, say, the addition of integer numbers up to 20. (Hence the name 'exercise' for a single 'sum'; once you arrive at the stage where you are going to test [the tables of] multiplication, division and, later still, fractions and beyond, the word 'sum' does not cover it anymore, as these are not 'sums' (additions) but rather different calculus exercises.

Preset

A predefined style, or set of styles, with optional extra parameters. 'Presets' are a technical solution, meant to help users of this calculus system produce exercise sheets (a.k.a. 'chapters') quickly and with maximum ease. Hence quite a few presets for 'first graders' (and several for students beyond that stage and age) are provided as one-click selections in the generator-driving web form.

Be aware that a 'preset' can contain one or more 'styles' and can optionally contain additional parameters to drive the calculus exercise generator. See the technical documentation if you wish to know more about presets, styles, their use and operation.

How does it work?

This section shows you several examples of use/try, where the examples slowly increase in complexity, starting with he most basic, introducing you to the various features of the calculus exercise generator step by step.

Following through ('replaying') an example takes a few minutes only, and it is very useful to try a few (preferably: all of them) for you to get a feel for the way the system works and what it can do for you while spending the least amount of time on it.

A basic example

Goal & Output

This example will produce a very basic set of simple calculus exercises for primary school children (age 6 years†) and their parents and teachers.

This example takes you through the (web) form and resulting page(s) of the calculus generator, in small, easy steps, showing you the basic use of the generator and its output(s).

We advise you to also try the next few examples, which slowly build up in complexity, both in features, use and results.

Execution

  • Go to the web form here (opens in another window / tab in your browser).

  • Go to the section titled Select preset(s) and submit by either clicking on the link in the table of contents at the top of that page or by scrolling down.

  • (Re-)set the 'Chapter' number to 1 (it should already show this value).

  • Then click on the item called 'simple additions in the range 1..10' in the (scrollable) preset pick list. This will be the second item in that list. Once clicked, this item will be 'marked' and consequently show as a line with a blue (i.e. 'selected' background).

  • Scroll down until you see the button titled 'OK to Generate'. (Keep the item values in the 'Page Setup' section as is: do not change those.) Click on the button ('OK to Generate') to submit your request to the web server; wait a little while for the generated exercise page to appear.

  • Now you can decide to print this page (and clip off the bottom section for perusal by a teacher or parent) or redo this action with the intent to only generate the exercises this time. Without the answers. That is going to be introduced in our next example too.

Results and Notes

Note that the exercise page includes both the exercises (4 columns of them) and the answers to those exercises a little further down on the same page; one answer for each of the exercises.

This is because the default Page Setup configuration is to:

  1. Try to generate 64 calculus exercises on a single page. (Some presets/settings may result in fewer exercises; see the technical documentation for more info about and ways to 'get around' this.)

  2. Print the exercises in a 4-column format, so the exercises are nicely spread across the page.

  3. Print/Show the generated exercises themselves.

  4. Print/Show the corresponding answers.

Technical Documentation

This part describes all the commands and options available with the calculus generator. Some examples are included to enhance understanding of the subject matter.

Interfacing

The calculus generator can be accessed using GET and/or POST web requests; if you need examples of both, you might wish to have a look at the HTML sources for the generic web form here and here and the links to the calculus generator PHP page in the education calendar. Otherwise also have a look at the info provided by Wikipedia.

When issuing a GET request, you should concatenate your selected set of tags (see table below for a list of those) according to RFC3986. See also Wikipedia for more info.

When using a POST request, you should adhere to RFC2616. (Wikipedia) Some features (config file upload) are only possible using HTTP POST. Make sure you encode the posted data using the multipart/form-data MIME type as specified by the W3C and RFC2388.

Do not forget to include the two square brackets in the tag: PresetCfg[]

Without those square brackets, you will only be able to request one(1) preset per request, while, when you include those square brackets, you will be able to request a mix of multiple presets by listing each preset in the requested mix as a separate PresetCfg[] item.

CustomConfigText is a tag specifically designed to allow sending custom configurations in requests to the calculus generator using <textarea> HTML form fields like the one in this example:

<textarea name="CustomConfigText" cols="128" rows="24">

WARNING

Be aware that URL query strings are quite limited in size, so submitting a multiline custom configuration using the CustomConfigText tag in a HTTP GET web form may be rejected when the custom configuration and other tag=value combinations surpass the URL query limit in length.

Hence we advise you to employ the CustomConfigText tag in HTTP POST based web forms only to prevent user confusion due to rejects by the web server as query string limitations are triggered.

Notes

The table above lists the web form tags in the order in which these are processed by the calculus generator web form interface.

Behind the scenes, these URL query string tags are all converted to equivalent config file lines, where the uploaded ConfigFile file contents and the CustomConfigText field value (i.e. the <textarea> content) are copied verbatim.

Also note that all tags are treated as optional items at this conversion stage and the combined results are concatenated, before the result is fed to the calculus generator proper. This behaviour implies that you can create web forms where you offer both custom config inputs ConfigFile and CustomConfigText at the same time. This is a legal use of the web form interface; when a user specifies content for both (i.e. specifies a config file for upload and enters one or more lines in the CustomConfigText field, both bits of content are concatenated in the order as shown in the table above, which means the CustomConfigText content will follow the content of the uploaded config file. See here for such a web form.

See the documentation about the config file format below for more info about the various tags and their precise behaviour.

The presets

The preset: config field (and the PresetCfg[] URL query string tag) can specify a single preset ID. When you upload a custom config file, you can specify your own additional presets between any set of styles_def_start: and styles_def_end: marker lines (see also the config file format description below), but the standard set of presets available with the calculus generator is available here.

Obscure Information

This preset textfile is always loaded by the calculus generator before your (custom) config/request data is processed. Hence you cannot override these presets as with presets the rule applies: first come is first served. I.e. any two presets, both called 'AA', will only have the first of these used throughout the config file parse process, thus 'hiding' the second preset definition.

The config file format

The config/instruction file is a multiline ASCII text file, where each line must match any of the following:

config line formats
Format Notes 
# comment line

Any line starting with a hash (#) will be ignored. 

An empty line is ignored too.

 
global_option: value

Specifies the value for a 'globally applied' setting, where 'global' means the setting is applied to all styles defined directly or indirectly in this config file. ('Indirect' style definitions are style definitions resulting from preset expansion.)

Obscure Information

Given multiple occurrences of a certain global option throughout the (expanded) config/instruction file, the following rule applies: the last active occurrence of each global option applies. Here we mean with active that only those global option lines are regarded which remain after all presets have been expanded.

An example may help. This completely fictious example:

styles_def_start:
A1
glob_opt1: A
---
A2
glob_opt1: B
---
B3
preset: A2
---
styles_def_end:
glob_opt1: C
preset: B3
glob_opt1: D

# B3 leads to A2 leads to this:
glob_opt1: B

glob_opt1: D

... which will be parsed as if this had been written instead:

# last occurrence wins:
glob_opt1: D

See also the obscure info bit at the styles_def_start: / styles_def_end: marker set below.

 
cmd options

Specifies a single style of exercises to be generated, for example a style which instructs the calculus generator to construct adding exercises (cmd=add), where, of course, the additional options apply:

# simple additions in the range 1..10
add lh_low=1 lh_high=10 rh_low=1 rh_high=10 


styles_def_start:
preset_definition
preset_definition
...
styles_def_end:

The set of styles_def_start: and styles_def_end: marker lines signals the calculus generator config/instruction parser that a series of preset definitions will follow. Such a section may contain zero or more presets, each of which are formatted like this:

preset_ID
config_line
config_line
...
---

where preset_ID is any non-whitespace character sequence, preferably consisting of alphanumeric and/or numeric characters. The first preset_ID which is identical to the value following a preset: global option is considered a match, resulting in all config_lines up to, but not including, the terminating triple dash --- being copied verbatim, replacing the preset statement in the config/instruction file.

Note that preset definitions can contain preset: option lines, thus allowing the nested use of presets.

Given this completely fictious config/instruction file snippet:

The global options

All global_option lines start with a word (alphanumerical, numerical and other non-whitespace characters combined), terminated by a ':' colon:

description:

All subsequent lines are treated as a description to be displayed. The description ends when a single '+' plus or '=' equals sign is located on a line.

When terminated by a '+' plus, the description will be appended to any previous description (use this with preset descriptions, so their descriptions concatenate when the user specifies multiple presets in a single run).

When terminated by an '=' equals sign, the description test will replace any previous description. (Use this to override (concatenated) preset descriptions.)

Caveat

Currently, descriptions are not used / displayed anywhere.

We are considering allowing arbitrary HTML formatting in the multiline description: text, which will be copied verbatim to the generated page...

The style commands (cmd) / Style Definitions

A style definition is a single line, which defines one particular set of tests and ranges, which will produce the desired exercises.

A style definition line is a collection of words, each separated by whitespace. Some may have an argument, separated by an '=' assignment character; the '=' may be surrounded by whitespace.

The first word of a style line is the 'operator' definition, i.e. the type of operation we want exercises to be generated for.

Operators:

(div) a divisor of 1 is rejected

(div with remainder) do prevent the occurrence of a zero remainder, i.e. a non-existing/empty remainder

 
nofrac1

(fraction calculus) do not accept a divisor of 1 for any fraction

 
units:

this attribute discards adherence to the lower and upper bound limitations for the terms it applies to. If it wouldn't lift those two restraints on the value, the 'unit' option would not be able to work for any exercises where you wish to combine unit-valued terms with higher numbers for other left hand terms.

 
asc, desc:

subtraction, division and hence any of their derivatives are not commutative, so (re-)ordering left hand term values using asc/desc will very probably have disastrous results, most often leading to rejection of an increasing number of attempts to produce a valid exercise. We strongly urge you not to use these two options with sub, div or any operation derived thereof.

 
T

The 'T' tables of multiplication subsetting lists the mandatory tables of multiplication as numbers, separated by ','.
