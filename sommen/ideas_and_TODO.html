ingeklede vraagstukken?

Dit is een heel ander chapiter; ingeklede vraagstukken vragen veel meer 'template' voorbereidingswerk: de sommen liggen daar wel aan ten grondslag, maar een 'ingekleed vraagstuk', hoe eenvoudig ook, is veel meer dan de som alleen

Bij ingeklede vraagstukken moeten de waarden ook enige relatie hebben met de realiteit (voorbeeld: iemand loopt geen 21 km/uur, maar misschien 5 of 7 als 'ie heel hard opschiet)

Daarbij komt ook nog eens dat (moeilijker?) ingeklede vraagstukken ook irrelevante informatie bevatten; een deel van de opgave is dan de juiste informatie uit de data te plukken.

ID code printed on every page

A unique ID code is printed on every page (exercise, answers, def) and is all you need to recreate the exercise it was printed on.

Since the number of permutations of permissible parameters is huge, there's no other way than to log generated exercises and assign them inque numbers that way; the alternative, generating a (reversible) hash value of the parameters used, is technically feasible, but would generate very long numbers which can be hard to copy by hand; a creditcard-like checksum would prevent most manual errors from slipping through, but the number would still be rather large as it would be a (compressed) serialization of the parameters used to generate the exercise.
